# CLAUDE.md

This file provides repository guidance to coding agents working in this project.

## Preferred agent workflows（代理工作流偏好）

When a task matches an available **gstack** skill or a **Compound Engineering** skill (`ce:*`), agents should **use those skills by default** rather than ad-hoc workflows.

- Prefer **gstack** skills for browser QA, design review, debugging, investigation, shipping, deploy verification, and related agent-driven workflows.
- Prefer **Compound Engineering** skills for planning, implementation workflow, code review, learnings capture, and other `ce:*` processes.
- If both are applicable, use the minimal combination that fits the task and follow their documented sequencing.
- If no relevant skill exists, continue with the best local workflow and state the fallback briefly.

## What this is

KS2 Mastery is a Cloudflare Workers web app for KS2 (Year 5/6, UK) study.
The frontend is a Vite-bundled React single-page app served alongside a Worker
API and D1-backed persistence layer.

Spelling is the real product surface today. Arithmetic, reasoning, grammar,
punctuation, and reading still share the production shell but mostly present
placeholder learning flows.

## Repository shape

- `index.html`
  Canonical browser entry shell. Loads content globals first, then the Vite
  module entry.
- `src/main.jsx`
  Frontend module entry. Imports the generated legacy bundle entry.
- `src/generated/legacy-entry.generated.jsx`
  Generated by `scripts/build-public.mjs`. Do not hand-edit it.
- `scripts/build-public.mjs`
  Generates the frontend entry, runs the Vite production build, and atomically
  swaps the final static output into `dist/public/`.
- `scripts/assert-build-public.mjs`
  Verifies the generated frontend contract and atomic-swap invariants.
- `worker/index.js`
  Worker composition root. Wires middleware, mounts route groups, exports the
  Durable Object, and delegates the asset path.
- `worker/lib/store.js`
  D1 persistence helpers plus migration-readiness checks. The Worker no longer
  creates tables at runtime.
- `docs/d1-migrations.md`
  Source of truth for local and remote schema workflow.
- `docs/operations-runbook.md`
  Current observability and incident-response baseline.
- `archive/KS2 Unified.html`
  Legacy snapshot kept for historical reference only. It is not the live
  delivery path.

## Local workflow

Install dependencies first:

```bash
npm ci
```

Recommended day-to-day commands:

```bash
npm run dev
npm test
npm run test:e2e
npm run check
npm run verify
```

What they do:

- `npm run dev`
  Builds the frontend, applies local D1 migrations, runs the local backfill,
  then starts `wrangler dev --local`.
- `npm test`
  Rebuilds `dist/public`, asserts the static output contract, then runs Vitest
  in the Workers runtime.
- `npm run test:e2e`
  Picks a free local port, starts the full local stack through
  `scripts/run-local-dev.mjs`, then runs Playwright.
- `npm run check`
  Rebuilds the frontend, asserts the static output contract, and performs a
  Cloudflare dry-run deploy.
- `npm run verify`
  Runs the current end-to-end verification sequence:
  `npm test && npm run check && npm run test:e2e`.

## Frontend delivery model

The production frontend is a bundled SPA, but the content globals are still
load-bearing.

Current order:

```text
vendor/sentence-bank-01.js … sentence-bank-06.js  -> banks on window
vendor/word-list.js                               -> window.KS2_WORDS
vendor/word-meta.js                               -> window.KS2_WORD_META
src/main.jsx                                      -> Vite module entry
```

Important rules:

- Never hand-edit `dist/public/`.
- Never hand-edit `src/generated/legacy-entry.generated.jsx`.
- If you add, remove, or reorder frontend source files in the generated legacy
  bundle, update the `clientSourceFiles` list in `scripts/build-public.mjs`.
- Keep `index.html`, `scripts/build-public.mjs`, and relevant tests aligned
  whenever the boot contract changes.

## Data and state

Server-side state lives in D1:

- users and linked identities
- sessions
- child profiles and child state
- spelling sessions
- subscriptions
- rate-limit buckets introduced by migrations

Migration discipline matters:

- `migrations/` is the schema source of truth.
- `worker/lib/store.js` verifies that every file listed in
  `EXPECTED_MIGRATIONS` has been applied.
- If you add a migration file, update `EXPECTED_MIGRATIONS` as part of the same
  change.

Browser-local state is intentionally small:

- `ks2-route` and `ks2-tab` keep shell navigation state in `localStorage`
- some TTS browser preferences remain local
- account, child, spelling, monster, Turnstile, and provider availability state
  now come from Worker APIs instead of local-only persistence

## Worker layering (`worker/`)

The Worker is layered as **composition root → middleware → routes → services → lib**.
There is deliberately no repository layer. If cross-cutting persistence
concerns appear later, add a repository layer when there is real logic to hold.

- **Composition root (`worker/index.js`)** — wires `instrumentRequest` and
  `ensureApiSchema`, mounts route groups, exports `SpellingLockDO`, and
  delegates asset serving.
- **Middleware (`worker/middleware/`)** — request context, schema readiness,
  and session extraction.
- **Routes (`worker/routes/`)** — Hono route groups, one per domain. Parse via
  contracts, delegate to services, serialise via contracts. No business logic.
- **Services (`worker/services/`)** — orchestration and business rules
  including auth, bootstrap, children, spelling, TTS, and bundle patching.
- **Contracts (`worker/contracts/`)** — request parsing and response envelope
  construction with runtime shape assertions.
- **Lib (`worker/lib/`)** — shared primitives: persistence, validation,
  observability, OAuth, Turnstile, TTS, rate limiting, session-cookie helpers,
  and domain constants.
- **Durable (`worker/durable/`)** — `SpellingLockDO` serialises mutations to a
  child's spelling session and learning state.

## Testing

The layered test strategy is documented in `docs/test-strategy.md`.

At a high level:

- `test/unit/`
  Pure module coverage for store, security, OAuth, migrations, observability,
  TTS normalisation, and spelling-domain logic.
- `test/integration/`
  Worker route coverage using `SELF.fetch(...)` in the Workers test runtime,
  including auth security, rate limits, observability, spelling flows, and TTS
  proxy behaviour.
- `test/e2e/`
  Browser flows through Playwright against the same local build-and-serve path
  used by developers.

When changing Worker request/response contracts, persistence, auth, security,
or delivery/build behaviour, update or add the corresponding automated
coverage.

## Security and operations

Current production baseline includes:

- Turnstile-gated register, login, and OAuth start flows
- request-rate limiting for auth and paid TTS endpoints
- server-side TTS provider handling
- `/api/health`
- `x-request-id` on responses
- structured Worker logs and an operations runbook

Before changing auth, migrations, health checks, logging, or D1 backup and
restore flows, read the relevant docs in `docs/`.

## Spelling and monster domain

The spelling pipeline flows: `SpellingGame` (UI) → `SpellingEngine` (grading +
spaced-repetition) → `MonsterEngine` (milestone detection) → `MonsterOverlay`
(celebration). Each word tracks a stage (0–6); correct answers move +1, wrong
−1. `SECURE_STAGE = 4` is the mastery threshold — reaching it for the first
time feeds the associated monster pool.

Each monster belongs to a subject and a word pool (`'y3-4' → inklet`,
`'y5-6' → glimmerbug`). `MonsterEngine.recordMastery` dedupes by word slug and
emits at most one event per transition:

| Event | Trigger |
|---|---|
| `caught` | 10 words mastered |
| `levelup` | every 10 after that |
| `evolve` | 50 / 80 mastered |
| `mega` | 100 mastered |

Thresholds live in `monsters.jsx` (`stageFor`, `levelFor`). Monster art is
inline hand-drawn SVG in `monsters.jsx`. `MonsterEngine.resetAll(pid)` exists
as a dev reset helper.

## Host edit mode

`src/app.jsx` exposes a host edit-mode protocol:

- posts `__edit_mode_available` on mount
- listens for `__activate_edit_mode` / `__deactivate_edit_mode`
- mirrors tweak edits back through `__edit_mode_set_keys`

Keep the `/*EDITMODE-BEGIN*/ ... /*EDITMODE-END*/` markers around `DEFAULTS`
intact. The external edit tool rewrites only the content inside that block.

## Design tokens and subjects

`TOKENS` in `src/tokens.jsx` is the source of visual truth for the shared shell.
`SUBJECTS` and `SUBJECTS_VIVID` define accent palettes and icon identities for
the six subjects. Read these before introducing hard-coded colours or new
families.

## Practical guardrails

- Treat `index.html` plus the Worker asset binding as the live delivery path.
- Treat `archive/` as reference material, not an active runtime.
- Keep generated files and build outputs reproducible; edit source files, not
  generated artefacts.
- If a change touches migrations, build contracts, or Worker runtime readiness,
  run the relevant verification commands before handing off.

## Skill routing

When the user's request matches an available skill, ALWAYS invoke it using the Skill
tool as your FIRST action. Do NOT answer directly, do NOT use other tools first.
The skill has specialized workflows that produce better results than ad-hoc answers.

Key routing rules:
- Product ideas, "is this worth building", brainstorming → invoke office-hours
- Bugs, errors, "why is this broken", 500 errors → invoke investigate
- Ship, deploy, push, create PR → invoke ship
- QA, test the site, find bugs → invoke qa
- Code review, check my diff → invoke review
- Update docs after shipping → invoke document-release
- Weekly retro → invoke retro
- Design system, brand → invoke design-consultation
- Visual audit, design polish → invoke design-review
- Architecture review → invoke plan-eng-review
- Save progress, checkpoint, resume → invoke checkpoint
- Code quality, health check → invoke health
